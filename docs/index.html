What is DST?

DST (Declining Stress Threshold) is a framework that shows how systems degrade over time, why they appear stable while getting worse, and when they will stop being able to handle change.

Instead of asking "Does this work?" — DST asks "Is this system getting stronger or weaker over time?"

The Model
ΘThetaReal remaining capacity
σSigmaStress that amplifies problems
κKappaMasking that hides damage
ρRhoReal healing
dΘ/dt = ρ − σ_eff
ρ > σ → system improves  ·  ρ < σ → declining (even if still working)

async function processPayment(userId, amount) {
  const result = await chargeAPI(userId, amount);

  if (!result.ok) {
    logger.error('Payment failed', {   // ρ
      userId, amount,
      code: result.errorCode,
      context: 'checkout_flow'
    });
    throw new PaymentError(             // ρ
      result.errorCode, { userId }
    );
  }
  return result.data;
}

ρExplicit throw — failure surfaces immediately. Structured log gives traceable context. Every caller gets a contract: success or typed exception. Gap closed.

async function loadUserProfiles(userIds) {
  if (userIds.length === 0)         // ρ guard
    return [];

  const response = await fetch(    // ρ single
    '/api/users/batch', {           // request
    method: 'POST',
    body: JSON.stringify({ ids: userIds })
  });

  if (!response.ok)
    throw new UserFetchError(       // ρ explicit
      { count: userIds.length }
    );
  return response.json();
}
σN+1 amplifier — stress scales with data size. Same code that works with 5 users catastrophically fails at 500. This is σ: the same pattern gets worse as load grows.
ρBatch request — O(1) requests regardless of input size. Error handling explicit. Θ stays constant as data scales 100×. σ_eff → 0.

04 React hooks — the dependency lie κ_a accumulated
BEFORE — κ_a dominant Θ = 18/100 
function UserProfile({ userId }) {
  const [user, setUser] = useState(null);

  useEffect(() => {
    fetchUser(userId).then(setUser); // κ_a
  }, []); // κ_a: empty array LIES
          // userId changes → stale data
          // fetch fails → silence
          // unmount mid-fetch → memory leak

  return <Profile user={user} />;
  // user=null on first render → crash
}
AFTER — ρ healing Θ = 85/100 
function UserProfile({ userId }) {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState(null);

  useEffect(() => {
    if (!userId) return;
    let cancelled = false; // ρ: cleanup flag
    setLoading(true);
    fetchUser(userId)
      .then(d  => { if (!cancelled) setUser(d); })
      .catch(e  => { if (!cancelled) setError(e); })
      .finally(() => { if (!cancelled) setLoading(false); });
    return () => { cancelled = true; }; // ρ: cleanup
  }, [userId]); // ρ: honest dependencies

  if (loading) return <Spinner />;
  if (error)   return <ErrorView error={error} />;
  return <Profile user={user} />;
}
κ_aEmpty dependency array — lies about what the effect depends on. userId changes, component shows stale user. Works in every demo. Fails in every production app with navigation.
ρHonest deps + cleanup flag — effect re-runs when userId changes. Cancelled flag prevents stale state on fast navigation. Error state surfaces failure explicitly. Three gaps closed at once.
↑ Θ: 18 → 85  ·  +67 pts  ·  three κ_a patterns stacked: dependency lie + missing cleanup + silent catch — all eliminated together
05 Caching — the σ disguised as optimization σ + κ_a
BEFORE — σ + κ_a Θ = 8/100 
const userCache = {}; // σ: grows forever

async function getUser(userId) {
  if (userCache[userId]) {
    return userCache[userId]; // κ_a: stale data
    // deleted user? still here
    // updated name? old name shown
    // no TTL, no invalidation
  }
  const user = await fetchUser(userId);
  userCache[userId] = user; // σ: permanent entry
  return user;
}
// month 1:  200 entries, fine
// month 6:  50,000 entries
// memory climbs, no one knows why
AFTER — σ resolved + ρ Θ = 91/100 
import LRU from 'lru-cache';

const userCache = new LRU({
  max: 500,              // ρ: bounded forever
  ttl: 5 * 60_000,      // ρ: 5min max stale
});

async function getUser(userId) {
  const cached = userCache.get(userId);
  if (cached) return cached;

  const user = await fetchUser(userId);
  userCache.set(userId, user);
  return user;
}

function invalidateUser(userId) {
  userCache.delete(userId); // ρ: explicit path
}

// month 1:  max 500 entries
// month 6:  still max 500 entries
// memory stable, data fresh
σ+κ_aUnbounded cache + stale data — the worst combination. σ grows memory proportionally to unique users forever. κ_a returns wrong data silently. Both feel like features ("it's fast!") while structurally degrading the system.
ρLRU + TTL + invalidation — memory bounded permanently. Data freshness guaranteed by TTL. Explicit invalidation path for immediate updates. Three structural guarantees in 4 lines.
↑ Θ: 8 → 91  ·  +83 pts  ·  lowest starting score of all 6 examples — σ+κ_a together is the most structurally damaging combination in any codebase

async function initDashboard(userId) {

  logAnalytics('opened', userId)   // ρ: handled
    .catch(e => logger.warn('analytics', e));

  const [user, prefs, notifs] =    // ρ: parallel
    await Promise.all([             // 200ms total
      fetchUser(userId),            // not 600ms
      fetchPrefs(userId),
      fetchNotifs(userId),
    ]);

  const data = await Promise.race([ // ρ: bounded
    fetchData(userId),
    new Promise((_,r) =>
      setTimeout(() => r(
        new TimeoutError({ ms: 3000 })
      ), 3000)
    )
  ]);

  return { user, prefs, notifs, data };
}
σ+κ_aSequential + fire-forget + no timeout — 600ms that grows with every added call. Analytics failures invisible. Dashboard can hang for minutes. Three completely separate structural problems in 15 lines.
ρPromise.all + .catch + Promise.race — 200ms regardless of call count. Analytics handled without blocking. 3 second ceiling on data fetch. σ eliminated, both κ_a gaps closed. Same data, 3× faster, bounded failure.

κ is not scalar — it is a vector

Not all masking is neglect. DST without this distinction becomes "refactor everything" — which is impossible. Some displacement is necessary. The difference is whether it is conscious.

    throw err; // ρ: visible failure
  }

// What AI writes with the circuit breaker rule
async function callService(data) {
  if (!circuit.isHealthy()) {   // ρ: check first
    return {
      status: 503,
      message: 'Service shedding load',
      retryAfter: circuit.resetIn() // ρ
    };

  try {
    return await circuit.call(     // ρ: bounded
      () => service.call(data),

    circuit.recordFailure();       // ρ: visible
    throw new ServiceError(err);  // ρ: explicit
  }
}

Three things to understand after this page

1. Not all problems should be fixed.
Some must be contained, bounded, and documented. κ_i is not weakness — it is physics.

2. Stability is not proof of health.
If κ is high — stability = borrowed time. y(t) ≠ Θ(t).

3. Every system has a trajectory.
The only real question is: Is Θ increasing or decreasing?

The core insight
Systems do not fail when they break.
They fail when they can no longer absorb change.

# DST-Aware Coding Assistant
# Declining Stress Threshold — ρ heals · κ hides · σ kills

Before writing any code, classify the request:
- Is this healing (ρ) or masking (κ)?
- If κ: accumulated (fix it) / conscripted (accept it) / intentional (document it)?

Mandatory rules — never violate:
1. No silent catch blocks. Every error must throw, rethrow, or log structured + throw.
2. No implicit state mutation. Return new objects. Side effects must be explicit.
3. No unbounded collections. Every array/map that grows must have a max size or eviction.
4. No N+1 patterns. DB/API calls inside loops are σ amplifiers — batch them.
5. No TODO without a ticket reference and expiration condition.
6. No `any` type without a documented reason and upstream issue link.

For every function written, ask:
- Does this make dΘ/dt positive (ρ) or negative (κ)?
- Is failure visible or invisible to callers?
- Does this scale safely or amplify stress with data size?

Output format for code reviews:
κ_a (fix): [pattern] — [why it declines Θ] — [specific fix]
κ_c (mitigate): [pattern] — [domain constraint] — [instrumentation approach]
ρ (preserve): [pattern] — [why it heals Θ]
σ (resolve first): [amplifier] — [scale factor] — [batch/bound fix]
